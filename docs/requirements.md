# Seki Agent 需求文档

## 1. 背景

Seki Agent 是基于公司业务场景建设的内部 Agent 工具，当前已具备文档翻译、知识库问答、SPI log 解析、文件上传、版本差分比较等能力。

本次工程化改造目标是将当前原型项目逐步演进为可维护、可测试、可部署、可扩展的前后端分离系统。

## 2. 用户范围

首期用户范围：

- 公司内部多部门用户。
- 预估用户规模约 50 人。
- 登录方式为内部账号密码。

暂不纳入首期范围：

- 外部客户用户。
- 公司 SSO、LDAP、企业微信、钉钉等统一登录。

## 3. 首期 MVP 能力

首期必须保留并工程化改造的能力如下。

### 3.1 文档翻译

用户可以上传待翻译文件，系统完成翻译并提供结果文件。

需求：

- 支持上传文件。
- 支持查看翻译任务状态。
- 支持下载翻译结果。
- 用户必须选择目标语言，系统不提供默认目标语言。
- 首期支持 `.pptx`、`.xlsx`、`.docx`。
- 翻译任务应支持异步处理，避免长时间阻塞页面。

待确认：

- 单文件最大大小。
- 是否需要保留翻译历史。
- 是否需要支持术语表或公司专有词库。

### 3.2 知识库问答

用户可以基于公司业务文档进行问答，系统通过 RAG 返回答案。

需求：

- 支持基于本地知识库检索回答。
- 回答应尽量展示引用来源。
- 支持中文业务文档。
- 保留当前基于 LangChain/LangGraph 的核心能力，并逐步模块化。
- 首期提供独立的知识库问答 API，后续 Agent 工具复用该能力。
- 首期保存用户隔离的 conversation 和 message 记录。

待确认：

- 知识库是否按部门、项目、客户或车型隔离。
- 是否需要知识库管理页面。
- 是否需要文档版本管理。
- 是否需要答案评价、反馈和人工纠错。

### 3.3 SPI log 解析

用户可以上传 SPI log，系统解析并输出结果。

需求：

- 支持上传 `.log` 文件。
- 支持触发解析任务。
- 支持查看任务状态。
- 支持下载最终 Excel 解析结果。
- 解析逻辑应从 UI 层拆分为独立模块，便于单元测试。

待确认：

- 是否需要历史记录。
- 是否需要批量解析。

### 3.4 文件上传与 Workspace

用户可以上传业务文件，系统按用户隔离存储。

需求：

- 用户只能访问自己的文件，除非后续明确增加共享空间。
- 文件上传、下载、删除应通过后端 API 管理。
- 文件存储路径不应散落在业务代码中，应统一由配置管理。
- 首期单文件大小限制为 600MB。
- 首期采用后端流式落盘，不做前端分片上传。

待确认：

- 是否需要部门共享文件区。
- 文件是否需要过期清理策略。
- 是否需要文件病毒扫描或敏感信息检查。
- 后续是否需要大文件分片、断点续传和秒传。

### 3.5 版本差分比较

用户可以对版本或文件进行差分比较。

需求：

- 首期输入为两个 `.tar.gz` 版本包。
- 系统自动解压版本包。
- 系统自动比较解压后 `bin` 和 `lib` 目录下的二进制文件。
- 差分比较逻辑应拆分为独立服务模块。
- 差分输入、输出、错误处理需要标准化。

待确认：

- 是否需要前端可视化差分结果。
- 是否需要保存差分历史。

## 4. 非功能需求

### 4.1 并发与性能

首期目标：

- 支持约 50 名内部用户使用。
- 支持多用户同时登录、上传文件、发起问答和提交后台任务。
- 文档翻译、知识库入库、SPI log 解析等耗时任务应异步处理。

待确认：

- 预期同时在线用户数。
- 预期同时执行的后台任务数量。
- 问答接口可接受的响应时间。
- 翻译和解析任务可接受的完成时间。

### 4.2 可用性

首期部署方式：

- Docker Compose。
- 本机部署。

首期目标：

- 支持一键启动主要服务。
- 服务异常时有明确日志。
- 关键运行数据持久化，不随容器重启丢失。

暂不作为首期强制目标：

- Kubernetes。
- 多机高可用。
- 自动横向扩缩容。

### 4.3 安全

首期要求：

- 内部账号密码登录。
- 密码必须使用安全哈希存储。
- API Key、数据库密码等敏感配置通过环境变量管理。
- 用户 workspace 隔离。

待确认：

- 是否需要管理员角色。
- 是否需要操作审计日志。
- 是否需要密码复杂度规则。
- 是否需要登录失败锁定策略。

### 4.4 可测试性

目标：

- 核心业务逻辑从 UI 和框架代码中拆出。
- 每个核心模块可以独立单元测试。
- 后端 API 可以进行接口测试。
- 耗时任务逻辑可以脱离真实外部 API 进行 mock 测试。

首期重点测试模块：

- 用户认证。
- 文件管理。
- 文档翻译任务调度。
- 知识库问答服务封装。
- SPI log 解析。
- 版本差分比较。

## 5. 工程化目标

### 5.1 前后端分离

目标：

- 使用 FastAPI 建立后端 API。
- 使用前端应用替代 Streamlit 的正式用户界面。
- Streamlit 可在迁移期保留作为旧入口或对照工具。

### 5.2 模块解耦

目标模块：

- `auth`：用户、登录、权限。
- `files`：文件上传、下载、删除、用户 workspace。
- `rag`：知识库问答。
- `translation`：文档翻译。
- `spi`：SPI log 解析。
- `diff`：版本差分比较。
- `agents`：Agent 编排。
- `storage`：数据库、文件系统、向量库适配。
- `tasks`：异步任务。

### 5.3 配置与部署

目标：

- 使用统一配置管理。
- 区分开发、测试、生产配置。
- Docker Compose 支持本机启动。
- 运行时数据与源码目录分离。

## 6. 首期验收标准

首期工程化改造完成时，应满足：

- 后端可以通过 Docker Compose 启动。
- 支持内部账号密码登录。
- 支持文件上传、下载、删除。
- 支持知识库问答 API。
- 支持文档翻译任务 API。
- 支持 SPI log 解析任务 API。
- 支持版本差分比较 API。
- 至少核心 service 层具备单元测试。
- 有基础接口测试。
- 有清晰的本地启动文档。

## 7. 当前明确不做

除非后续另行确认，首期不做：

- 外部客户访问。
- 公司统一登录集成。
- Kubernetes 部署。
- 多机高可用集群。
- 复杂权限审批流。
- 完整知识库运营后台。
